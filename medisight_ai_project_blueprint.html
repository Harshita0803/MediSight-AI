Data flow in detail
Structured path (SQL + ML): Raw CSV (patient vitals, labs, diagnoses, medications) → ETL cleans, validates, transforms → loaded into PostgreSQL star schema (fact_encounters, dim_patient, dim_diagnosis) → feature engineering creates age_group, comorbidity_score, avg_lab_deviation → Scikit-learn + XGBoost pipeline trains readmission classifier and LOS regressor → model artifacts saved with joblib → FastAPI loads models at startup and scores on demand.

Unstructured path (RAG): Clinical notes (discharge summaries, progress notes) → text cleaning → chunked into 512-token overlapping segments → embedded using sentence-transformers (free, local) → stored in FAISS index or pgvector → LangChain retriever fetches top-5 chunks → system prompt + context + question → Groq LLaMA-3 (free API) generates answer → response includes cited source chunks.
Phase 1 — Data sourcing & ETL pipeline (weeks 1–2)
Step 1 — Get free data
Use two sources. First: register at physionet.org to access MIMIC-III demo (freely available, no cost — 100 de-identified patients with real clinical notes). Second: generate synthetic structured EHR data using Synthea (free Java tool from MITRE) — run locally, generates realistic CSV files for thousands of patients with diagnoses, labs, vitals, and medications.

Step 2 — Design the database schema
Build a Star Schema in PostgreSQL (free, run locally via Docker or install directly). Central fact table is fact_encounters — one row per hospital visit. Dimensions: dim_patient, dim_diagnosis (ICD codes), dim_medication, dim_lab_result. This is exactly the Star Schema work from your Crayon experience — point to it directly in your portfolio.
CREATE TABLE fact_encounters ( encounter_id VARCHAR PRIMARY KEY, patient_id VARCHAR REFERENCES dim_patient(patient_id), admission_date DATE, discharge_date DATE, length_of_stay_days INT, readmitted_30d BOOLEAN, -- ML target 1 primary_diagnosis_code VARCHAR ); CREATE TABLE dim_patient ( patient_id VARCHAR PRIMARY KEY, age INT, gender VARCHAR, race VARCHAR, insurance_type VARCHAR, zip_code VARCHAR );
Step 3 — Build the ETL pipeline
Write a Python ETL pipeline with four stages: Extract (read CSVs/JSON from Synthea output), Transform (clean nulls, standardize date formats, map ICD codes, calculate derived fields), Validate (data quality checks — null rates, referential integrity, value range checks), Load (bulk insert into PostgreSQL with psycopg2). Add logging at each stage. Structure as a proper Python module with config files.
project/ etl/ extract.py # reads raw Synthea CSV files transform.py # cleaning, standardization, feature calc validate.py # data quality checks (null %, ranges, FK integrity) load.py # PostgreSQL bulk insert via psycopg2 pipeline.py # orchestrates all stages, logs each step config/ db_config.yaml # DB connection (use .env for secrets) etl_config.yaml # chunk sizes, validation thresholds

After loading raw data, run a second transformation pass to create ML-ready features. This is your "AI-ready feature dataset" concept from Crayon. Phase 2 — ML models (weeks 3–4)
Model 1 — 30-day readmission classifier
Binary classification problem: will this patient be readmitted within 30 days? Use XGBoost (best in class for tabular healthcare data). Build a full Scikit-learn Pipeline with preprocessing (ColumnTransformer for numeric scaling + categorical encoding) and the model. Use stratified k-fold cross-validation. Tune with RandomizedSearchCV. Evaluate with ROC-AUC, F1, precision-recall curve. Handle class imbalance with class_weight='balanced'.

Phase 3 — RAG pipeline & LLM layer (weeks 5–6)
Step 1 — Prepare clinical text data
Use MIMIC-III demo clinical notes (discharge summaries, radiology reports, nursing notes). Pre-process: remove PHI placeholders, normalize whitespace, split by note type, tag each note with metadata (patient_id, note_type, admission_date, department). Store processed notes as JSON files with these metadata fields — this metadata becomes your retrieval filter.
note_record = { "patient_id": "10006", "note_id": "note_001", "note_type": "discharge_summary", "admission_date": "2150-03-12", "department": "cardiology", "text": "Patient is a 67-year-old male admitted for CHF exacerbation...", "word_count": 842 }
Step 2 — Chunking strategy
Use a hybrid chunking strategy: semantic chunking (split on sentence boundaries, not arbitrary token counts) with 512-token chunks and 64-token overlap. Add a metadata prefix to each chunk so the LLM knows context even if only given the chunk. This is the chunking design work you did at Humanity Labs — describe it exactly this way in your portfolio.
from langchain.text_splitter import RecursiveCharacterTextSplitter splitter = RecursiveCharacterTextSplitter( chunk_size=512, chunk_overlap=64, separators=["\n\n", "\n", ". ", " "] # semantic-first splitting ) def chunk_note(note: dict) -> list[dict]: chunks = splitter.split_text(note['text']) return [{ "chunk_id": f"{note['note_id']}_chunk_{i}", "text": f"[{note['note_type']} | {note['admission_date']}]\n{chunk}", "metadata": {k: note[k] for k in ['patient_id','note_type','admission_date','department']} } for i, chunk in enumerate(chunks)]
Step 3 — Embedding & vector store
Use sentence-transformers (100% free, runs locally) — specifically the model "all-MiniLM-L6-v2" (384 dimensions, fast, high quality for clinical text). Build both a FAISS index (for speed demos) and pgvector storage (for production demos) — having both lets you compare in your portfolio. Use metadata filtering to restrict retrieval to a specific patient_id.
from sentence_transformers import SentenceTransformer import faiss, numpy as np model = SentenceTransformer('all-MiniLM-L6-v2') # free, local, no API key # Embed all chunks texts = [c['text'] for c in all_chunks] embeddings = model.encode(texts, batch_size=64, show_progress_bar=True) embeddings = np.array(embeddings).astype('float32') # Build FAISS index dim = embeddings.shape[1] # 384 index = faiss.IndexFlatIP(dim) # inner product (cosine after normalize) faiss.normalize_L2(embeddings) index.add(embeddings) faiss.write_index(index, 'vector_store/clinical_notes.index')
